# web-splat Technical Notes

## Summary

web-splat is a compact Rust and WGPU Gaussian splatting viewer that pushes almost all heavy per-frame work into GPU compute. Its optimization profile is split across five layers: device bring-up, preprocess-and-pack, GPU radix sort, compressed data ingestion, and wasm/runtime delivery.

## Architecture Summary

- `src/lib.rs` owns WGPU initialization, surface configuration, feature negotiation, and runtime policy such as HDR and v-sync mode.
- `src/renderer.rs` owns the render pipeline, preprocess compute stage, indirect draw state, and sorter lifecycle.
- `src/gpu_rs.rs` and `src/shaders/radix_sort.wgsl` implement a GPU radix sort derived from the Fuchsia sorter design.
- `src/pointcloud.rs`, `src/io/ply.rs`, and `src/io/npz.rs` define the upload layout and compressed/uncompressed loading paths.
- `src/shaders/preprocess.wgsl` and `src/shaders/preprocess_compressed.wgsl` are the critical view-dependent transformation stage that turns 3D gaussians into screen-space splats.
- `build_wasm.sh` and `.github/workflows/viewer.yml` form the browser delivery path.

## Dominant Optimization Patterns

### GPU-First Visibility and Draw Control

The renderer projects, culls, packs, sorts, and counts splats on the GPU. The CPU prepares camera and render-setting uniforms, but visible-splat counting and indirect draw setup stay in GPU buffers.

### Radix Sort as a Reusable Primitive

The sorter is not just a utility; it is a central optimization surface. The implementation specializes workgroup constants, reuses scratch buffers, tests subgroup sizes empirically when needed, and hands sorted instance counts directly to the draw path.

### Aggressive Payload Compression

The codebase consistently prefers compressed or half-precision representations when possible: `f16` covariance, packed screen-space splat payloads, quantized SH coefficients, and shared index indirection for compressed data.

### Runtime Profiles by Target

Native builds request richer GPU features and profiling capability. Wasm builds use a smaller dedicated profile and strip the binary. Optional support such as NPZ input is feature-gated instead of always-on.

## Transferable Lessons

- A small renderer can still expose a rich optimization catalog when its abstractions are tight.
- If the render pass is simple, most optimization leverage lives in the compute stages before it.
- Loader and asset-layout choices are part of the performance model, not just I/O plumbing.
- Browser deployment should be treated as its own optimization tier.

# 3D Gaussian Splatting Renderer implemented in WebGPU (WGPU) and Rust

This code implements a renderer for the reconstructions obtained with [3D Gaussian Splatting](https://github.com/graphdeco-inria/gaussian-splatting)

[👉 Click to run the web demo 👈](https://keksboter.github.io/web-splat/demo.html)


## Build

First, install the Rust compiler (e.g. with [rustup](https://rustup.rs/)) 

Clone the repository and run

```bash
cargo build --release --bin viewer 
```

## Run

Use the `point_cloud.ply` and `cameras.json` files generated by [3D Gaussian Splatting](https://github.com/graphdeco-inria/gaussian-splatting):

```
cargo run --release --bin viewer point_cloud.ply cameras.json
```

To load [compressed npz files](https://github.com/KeKsBoTer/c3dgs) the `npz` feature must be enabled:

```
cargo run --release --features npz --bin viewer point_cloud.npz cameras.json
```

## About

**Splat Sorting**
: We ported the [Fuchsia RadixSort](https://fuchsia.googlesource.com/fuchsia/+/refs/heads/main/src/graphics/lib/compute/radix_sort/) to WGPU for sorting the splats on the GPU.

**Performance**: The renderer reaches >200 FPS on a `NVIDIA 3090 RTX` and ~130 FPS on a `AMD Radeon R9 380 Series` (8 years old). Measurements where taken for the bonsai scene at 1200x799 resolution.
